Culture as Composable Blocks: Configuring AI Context Beyond System Prompts

April 2026

TL;DR System prompts and CLAUDE.md files are primitive ways to configure AI behavior. What if we treated AI "culture" as a composable system — discrete knowledge blocks that merge to produce a coherent working context? I'm exploring a UI where you pick and combine blocks to shape how an AI agent thinks, writes, and decides.

Every team that works with AI coding agents eventually hits the same wall: the system prompt becomes a junk drawer.

You start with a clean instruction file. Then someone adds coding conventions. Then error handling preferences. Then the deployment process. Then a paragraph about "don't use semicolons." Six months later, you have a 2,000-line CLAUDE.md that no one fully reads — least of all the AI.

This is a culture problem disguised as a configuration problem.

What is culture, really?

In human organizations, culture isn't a document. It's the emergent result of overlapping knowledge systems: the company's values, the team's habits, the tech stack's constraints, the industry's norms, and each individual's experience. No single person holds all of it. It lives in the intersection.

When we configure AI agents today, we flatten all of this into a single text blob. We lose the structure. We lose the composability. And we lose the ability to reason about which parts of the culture matter for which context.

Knowledge blocks

Here's the idea: what if we decomposed AI context into discrete, reusable knowledge blocks — each representing one facet of culture?

Each block is a self-contained unit with:

  • A domain — what aspect of work it governs (style, testing, architecture, communication, security...)
  • Opinions — concrete preferences, not vague guidelines ("use Result types for errors" not "handle errors properly")
  • Tensions — explicit trade-offs ("we prefer readability over performance, except in hot paths")
  • Anti-patterns — what this culture actively rejects

Composition, not concatenation

The key insight is that merging blocks isn't just appending text. When two blocks have opinions on overlapping areas, the merge needs to resolve them. This is where it gets interesting.

Consider two blocks:

Block: "Move Fast"
- Ship MVP first, iterate later
- Prefer simple solutions
- Skip abstractions until the third use case

Block: "Production Grade"
- No shortcuts in error handling
- Every public API needs input validation
- Comprehensive logging on all state transitions

These blocks have a natural tension. A naive concatenation would give the AI contradictory instructions. But a proper merge produces something richer: "ship fast, but never skip error handling. Simple solutions are preferred, but they must be correct solutions."

This is exactly how real culture works. No team is purely "move fast" or purely "production grade." Culture is the specific resolution of tensions between competing values. The merge is the culture.

The UI I'm imagining

Picture a workspace configuration screen. Instead of editing a giant text file, you see a grid of available knowledge blocks — some provided by the community, some by your organization, some personal.

You drag blocks into your workspace. As you add them, a preview shows the merged culture — the resolved set of principles your AI agent will operate under. Conflicts are highlighted. You can adjust priority (which block "wins" on a given tension) or write explicit resolution rules.

Why this matters now

We're entering an era where AI agents don't just write code — they make architectural decisions, choose libraries, handle edge cases, and interact with production systems. The "culture" they operate under directly impacts the quality and safety of their output.

Right now, configuring that culture is like writing a constitution from scratch every time you start a project. Knowledge blocks would let us:

  • Share and reuse proven cultural patterns across teams and organizations
  • Version control cultural decisions independently (your security posture evolves separately from your style guide)
  • Compose context-appropriate cultures — your payments service gets different blocks than your internal admin tool
  • Make trade-offs explicit — no more implicit contradictions buried in a 2,000-line prompt
  • Onboard AI agents the way we onboard humans — gradually, with structured knowledge, not a wall of text

This already exists, partially

If you squint, you can see proto-versions of this pattern:

  • CLAUDE.md files are single, monolithic knowledge blocks
  • Cursor rules and .cursorrules files serve the same purpose
  • ESLint configs are composable style blocks (extend, override, merge)
  • Docker Compose files compose infrastructure "blocks"
  • Terraform modules compose infrastructure culture

The mental model exists in adjacent domains. We just haven't applied it to AI agent configuration yet.

Open questions

I don't have all the answers. Some things I'm still thinking about:

  • Granularity — how big or small should a block be? Too granular and you're back to configuration hell. Too coarse and you lose composability.
  • Conflict resolution — should it be manual (user picks the winner), algorithmic (priority scores), or AI-assisted (the agent itself resolves tensions)?
  • Evaluation — how do you measure whether a merged culture actually produces better output? What does "better" even mean here?
  • Drift — cultures evolve. How do you update blocks without breaking the merges that depend on them?
  • Context windows — every block consumes tokens. How do you keep the merged culture within practical limits?

What I'd love to hear

I'm sharing this because I think composable AI culture is an underexplored design space. If you're working with AI coding agents at any scale, I'd love to know:

  • How do you currently manage your AI agent's "culture"? What's working, what's not?
  • Have you seen this composability pattern emerge naturally in your workflows?
  • What would you want from a knowledge block system that a flat file can't give you?

The goal isn't to replace system prompts — it's to give them structure. The same way we went from shell scripts to infrastructure-as-code, maybe it's time to go from prompt-as-text to culture-as-composition.
