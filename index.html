Your AI Has No Culture

April 2026

When you hire someone, you don't hand them a 2,000-line document and say "this is how we do everything." They absorb culture over time — from the team, the codebase, the way people talk in code reviews, the unspoken rules about what matters and what doesn't.

But that's exactly what we do with AI agents. We write a giant instruction file — a CLAUDE.md, a system prompt, a .cursorrules — and dump everything in there. Coding style, error handling, deployment rules, architecture preferences, "don't use semicolons." One flat file. No structure. No priority. No way to say "this matters more than that."

And it kind of works, until it doesn't.

* * *

Here's what I keep noticing. Every team I've seen using AI coding agents ends up with the same problem: their instruction file becomes a junk drawer. It grows. It contradicts itself. Nobody remembers what's in it. The AI follows some rules and ignores others, and you can't tell why.

The deeper problem is that we're trying to encode culture as a document. And culture isn't a document. Culture is what happens when different ideas about how to work collide and get resolved.

Think about it. A team's culture is never just one thing. It's the intersection of: how we write code + how we handle risk + how much we value speed vs. correctness + what we've been burned by before. No single person holds all of it. It lives in the overlap.

* * *

So here's an idea I've been sitting with for a while.

What if, instead of one big file, we broke AI context into small, reusable pieces? I'm calling them knowledge blocks. Each block captures one opinion about one thing.

┌───────────────┐ ┌───────────────┐ ┌───────────────┐ │ "Code Style" │ │ "Error Policy" │ │ "Security" │ │ │ │ │ │ │ │ terse, │ │ fail loud, │ │ zero trust, │ │ functional, │ │ no silent │ │ validate at │ │ no classes │ │ catches │ │ boundaries │ └───────┬────────┘ └───────┬────────┘ └───────┬────────┘ │ │ │ └────────────────────┼─────────────────────┘ ▼ ┌───────────────────┐ │ merged = culture │ └───────────────────┘

Simple so far. But the interesting part isn't the blocks themselves — it's what happens when they overlap.

The merge is the culture

Say you have two blocks:

"Move Fast" — ship the MVP, skip abstractions, iterate later.

"Production Grade" — never skip error handling, validate everything, log all state transitions.

These two blocks contradict each other. If you just concatenate them, the AI gets confused. But if you resolve the tension — "ship fast, but never skip error handling; simple solutions are preferred, but they must be correct" — now you have something interesting.

You have a culture.

That's the key insight. Culture isn't any single value. It's the specific way a team resolves the tension between competing values. "Move Fast" alone is not a culture. "Production Grade" alone is not a culture. But "move fast AND production grade, and here's how we balance them" — that's a culture.

The merge is where the meaning lives.

* * *

What this could look like

Imagine a UI where you're setting up a new AI workspace. Instead of opening a blank text file, you see a library of knowledge blocks. Some are yours. Some are shared by your team. Some are from the community.

You pick the ones that apply. As you add blocks, the system shows you where they agree and where they conflict. You resolve the conflicts — either by choosing a priority, or by writing a short rule ("at service boundaries, security wins over simplicity").

The result: a merged culture, custom to this project, built from reusable parts.

Workspace: "payments-api" Blocks: [Go Style Guide] [Security First] [Team: Backend] ⚡ 1 conflict: "Go Style" → minimal error wrapping "Security First" → full error context always → Your resolution: wrap at service boundaries Merged culture: • Idiomatic Go, standard library preferred • All inputs validated at service boundaries • Errors wrapped with context at boundaries, lean elsewhere • ...

Different project? Different blocks. Your payments API gets "Security First." Your internal admin tool gets "Move Fast." Same team, different cultures, because the context is different. Just like real teams adjust how they work based on what they're building.

* * *

Why not just edit the file?

Because flat files don't scale, and they don't compose.

When your instruction file is 200 lines, it's fine. When it's 2,000 lines, it's not fine. You get contradictions nobody notices. You get rules that made sense for one project copy-pasted into another where they don't. You get context that's always loaded even when it's irrelevant.

Knowledge blocks solve this the same way functions solved copy-pasting code. You write the opinion once, reuse it everywhere, and when it changes, it changes in one place.

But more importantly: blocks make trade-offs visible. Right now, if your instruction file says "be fast" in paragraph 3 and "be thorough" in paragraph 47, the contradiction is invisible. With blocks, conflicts are surfaced. You have to resolve them. That resolution is a decision, and decisions are how culture gets made.

* * *

This pattern already exists

We already think this way in other domains. ESLint configs extend and override each other — they're composable style blocks. Docker Compose files merge service definitions. Terraform modules are reusable infrastructure opinions.

Even CLAUDE.md files have a proto-version of this: there's a global one, a project one, and they merge. But it's just two levels, with no conflict resolution, no visibility into what overrides what.

The mental model is there. We just haven't built the tool yet.

* * *

Open questions

I don't have a finished design. Some things I'm still thinking through:

How big should a block be? Too small and you're drowning in config. Too big and you lose the composability. My gut says: one block = one clear opinion about one domain. Like a function — it does one thing.

Who resolves conflicts? The user manually? An algorithm with priority scores? Or the AI itself? Maybe all three, depending on the stakes.

How do you know it's working? Culture is hard to measure even with humans. With AI, maybe we can actually test it — run the same task with different block combinations and see what changes.

Context window limits. Every block costs tokens. You can't load everything. So there's a natural pressure to keep blocks focused and to only load what's relevant — which is probably healthy.

* * *

I think we're at an early stage with AI agent configuration. The flat instruction file is our version of the shell script before infrastructure-as-code. It works, but it doesn't scale, and it doesn't capture what we actually mean.

What we actually mean is: "work like us." And "us" is not a document. It's a culture — a specific set of resolved tensions between things we care about.

If we can make those tensions composable, explicit, and reusable, we might finally have a real way to teach AI agents how we work.

I'd love to hear how you're handling this. What does your instruction file look like? What works, what's broken? Has anyone tried splitting it into pieces?
