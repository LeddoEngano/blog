Your AI Has No Culture

April 2026

Every team using AI coding agents ends up in the same place: a giant instruction file (CLAUDE.md, .cursorrules, system prompt) that started clean and became a junk drawer. Coding style next to deployment rules next to "don't use semicolons." It grows, contradicts itself, and nobody fully reads it. Including the AI.

The problem isn't the file. The problem is that we're trying to encode culture as a flat document. And culture isn't a document. Culture is what happens when different ideas about how to work collide and get resolved.

What if we broke that file into small, reusable pieces? I'm calling them knowledge blocks. Each block captures one opinion about one thing.

┌───────────────┐ ┌───────────────┐ ┌───────────────┐ │ "Code Style" │ │ "Error Policy" │ │ "Security" │ │ │ │ │ │ │ │ terse, │ │ fail loud, │ │ zero trust, │ │ functional, │ │ no silent │ │ validate at │ │ no classes │ │ catches │ │ boundaries │ └───────┬────────┘ └───────┬────────┘ └───────┬────────┘ │ │ │ └────────────────────┼─────────────────────┘ ▼ ┌───────────────────┐ │ merged = culture │ └───────────────────┘

The interesting part isn't the blocks. It's what happens when they overlap.

The merge is the culture

Two blocks: "Move Fast" (ship the MVP, skip abstractions, iterate later) and "Production Grade" (never skip error handling, validate everything, log state transitions). They contradict each other.

Concatenate them and the AI gets confused. But resolve the tension ("ship fast, but never skip error handling") and you get something real. You get a culture.

That's the key insight. Culture isn't any single value. It's the specific way you resolve the tension between competing values. "Move Fast" alone is not a culture. "Production Grade" alone is not a culture. The resolution between them is.

* * *

What this could look like

A UI where instead of editing a text file, you pick knowledge blocks from a library (yours, your team's, community-shared) and drop them into your workspace. The system shows where they conflict. You resolve the conflicts by setting priority or writing a short rule ("at service boundaries, security wins over simplicity").

Result: a merged culture, custom to this project, built from reusable parts.

Workspace: "payments-api" Blocks: [Go Style Guide] [Security First] [Team: Backend] ⚡ 1 conflict: "Go Style" → minimal error wrapping "Security First" → full error context always → Your resolution: wrap at service boundaries Merged culture: • Idiomatic Go, standard library preferred • All inputs validated at service boundaries • Errors wrapped with context at boundaries, lean elsewhere • ...

Different project, different blocks. Your payments API gets "Security First." Your internal admin tool gets "Move Fast." Same team, different cultures for different contexts. Like real teams already do.

* * *

Why not just edit the file?

Flat files don't compose. At 200 lines they're fine. At 2,000 lines you get contradictions nobody notices, rules copy-pasted from other projects where they don't apply, and context that's always loaded even when irrelevant.

Blocks solve this the way functions solved copy-pasting code. Write the opinion once, reuse it everywhere, change it in one place.

More importantly: blocks make trade-offs visible. If your instruction file says "be fast" in paragraph 3 and "be thorough" in paragraph 47, the contradiction is invisible. With blocks, conflicts are surfaced and you have to resolve them. That resolution is a decision, and decisions are how culture gets made.

* * *

This pattern already exists

ESLint configs extend and override each other. Docker Compose merges service definitions. Terraform modules are reusable infrastructure opinions. Even CLAUDE.md has a proto-version: global file + project file that merge. But it's just two levels with no conflict resolution.

The mental model is there. We just haven't built the tool yet.

* * *

Open questions

How big should a block be? One clear opinion about one domain. Like a function, it does one thing.

Who resolves conflicts? The user? An algorithm? The AI itself? Probably all three depending on stakes.

How do you test it? Run the same task with different block combinations and compare. Culture is hard to measure with humans. With AI we can actually A/B test it.

Context limits. Every block costs tokens. Natural pressure to keep them focused and only load what's relevant.

* * *

The flat instruction file is our shell script before infrastructure-as-code. It works but doesn't scale and doesn't capture what we actually mean. What we mean is: "work like us." And "us" is not a document. It's a set of resolved tensions between things we care about.

If we can make those tensions composable, explicit, and reusable, we have a real way to teach AI agents how we work.

How are you handling this today? What does your instruction file look like? Has anyone tried splitting it into pieces?
